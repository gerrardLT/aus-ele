## 1. 项目定位与目标

### 1.1 这是什么项目

`aus-ele` 是一个能源市场分析工作区，当前包含两条主线：

- 澳洲电力市场分析主线：围绕 AEMO 体系下的 NEM / WEM 数据，提供价格、事件、预测、套利、FCAS 和投资分析
- Fingrid 数据主线：围绕芬兰 Fingrid 数据集，提供独立的数据同步、聚合、展示和导出能力

它不是自动交易系统，也不是投资自动决策系统。它更接近一个“市场分析与解释平台”，用于把原始市场数据翻译成更可读的业务结论和工程信号。

### 1.2 项目想解决什么问题

从业务角度，这个项目主要解决四类问题：

- 市场观察：不同区域、不同年份、不同市场的价格结构、负价分布和波动形态如何
- 信号判断：当前是否存在套利窗口、FCAS 机会、电网风险或结构性事件
- 收益估算：在一组给定假设下，BESS 的套利、FCAS 和投资回报大致处于什么区间
- 数据产品化：把原始外部市场数据转成可查询、可展示、可复核、可部署的内部分析资产

### 1.3 当前真实边界

当前系统可以稳定承担的事情：

- 历史价格、事件和规则型预测的可视化与解释
- 面向 BESS 的机会分析、粗粒度收益估算和财务情景分析
- Fingrid 单数据集的同步、聚合、摘要、图表和导出
- 单机 SQLite + Redis 组合下的本地和轻量部署

当前系统不应被当成的东西：

- 生产级自动下单系统
- 多市场联合最优调度系统
- 经审计的投资决策引擎
- 对未来价格进行高置信度机器学习预测的系统

## 3. 系统总体架构

### 3.1 业务理解

系统整体上是一个“四层结构”：

1. 外部数据源层  
   AEMO、WEM、Fingrid 与相关公开数据源
2. 采集与存储层  
   脚本将数据抓取到本地 SQLite，部分热点接口配 Redis 缓存
3. 分析服务层  
   FastAPI 将原始数据转成面向图表和业务分析的 API 结果
4. 展示与交互层  
   React 前端按模块消费 API，形成主工作台和独立页面

### 3.2 工程实现

主要数据流如下：

1. `scrapers/` 脚本从外部抓取数据
2. `backend/database.py` 管理表结构、写入和读取
3. `backend/server.py` 暴露 API
4. 前端通过 `fetch` 请求 `/api/*`
5. 组件按业务模块展示结果
6. 高频接口使用 Redis 做响应缓存
7. 后端内置调度器可以定时触发市场同步与 Fingrid 增量同步

## 4. 数据源与采集链路

## 4.1 澳洲市场数据

### NEM

来源：

- AEMO NEM 历史价格归档
- NEM FCAS 相关价格字段

主要脚本：

- `scrapers/aemo_nem_scraper.py`

输出特点：

- 以年分表形式写入 `trading_price_YYYY`
- 除能量价外，还包含 NEM FCAS 价格列

### WEM

来源：

- WEM Reference Trading Price
- WEM dispatch solution 派生出的 ESS slim 数据

主要脚本：

- `scrapers/aemo_wem_scraper.py`
- `scrapers/aemo_wem_ess_scraper.py`

输出特点：

- WEM 能量价写入 `trading_price_YYYY`，区域标识为 `WEM`
- WEM ESS slim 写入独立表，而不是复用 NEM FCAS 结构

### 电网事件

来源：

- 市场公告、停运信息、调度相关事件源

主要脚本：

- `scrapers/aemo_grid_event_scraper.py`

用途：

- 为 `event-overlays` 和 `grid-forecast` 提供解释层与风险驱动

## 4.2 Fingrid 数据

来源：

- Fingrid Open Data API

主要脚本：

- `scrapers/fingrid_sync.py`

后端支持：

- `backend/fingrid/client.py`：HTTP 客户端
- `backend/fingrid/service.py`：同步编排、查询、聚合
- `backend/fingrid/catalog.py`：支持的数据集目录

当前 README 中重点示例是 dataset `317`。

## 4.3 采集策略与现实约束

- 澳洲市场数据以本地持久化为主，适合做离线分析和增量更新
- Fingrid 依赖 API key，且存在节流、超时和时间窗限制
- WEM ESS 当前是 slim 表路线，强调轻量预览而非全量市场还原
- 该项目的数据采集设计重点在“可维护”和“可部署”，而不是全量历史仓库重建

## 5. 数据库存储与表结构

### 5.1 存储总览

项目当前核心存储是 SQLite，本地主文件通常位于：

- `data/aemo_data.db`

它承载三类数据：

- 市场原始 / 半原始时序数据
- 分析缓存与同步状态
- Fingrid 数据目录与时序数据

### 5.2 澳洲市场主价格表

结构特点：

- `trading_price_2022`
- `trading_price_2023`
- `trading_price_2024`
- `trading_price_2025`
- `trading_price_2026`

设计含义：

- 按年分表，降低单表规模压力
- NEM 和 WEM 共用主价格表体系，但通过 `region_id` 区分市场
- NEM 表含 FCAS 字段，WEM 同表行则主要承载能量价

### 5.3 WEM ESS slim 表

当前 WEM ESS 相关表包括：

- `wem_ess_market_price`
- `wem_ess_constraint_summary`
- `wem_ess_capability`

设计含义：

- 不追求保存原始超大 JSON
- 只保留 FCAS 预览、约束和能力分析必需的瘦字段
- 主要服务于 WEM FCAS 分析和 WEM 预测能力

### 5.4 事件与预测相关表

- `grid_event_raw`
- `grid_event_state`
- `grid_event_sync_state`
- `grid_forecast_snapshot`
- `grid_forecast_sync_state`

设计作用：

- 事件归一化与状态化
- 预测快照缓存
- 同步状态追踪

### 5.5 Fingrid 相关表

- `fingrid_dataset_catalog`
- `fingrid_timeseries`
- `fingrid_sync_state`

设计作用：

- 保存支持的数据集目录
- 保存时序数据
- 保存同步进度、错误和覆盖范围

### 5.6 分析缓存与系统状态

- `analysis_cache`
- `system_status`

说明：

- `analysis_cache` 用于投资分析等较重计算的结果复用
- `system_status` 用于记录系统级状态，例如最后更新时间

### 5.6.1 数据质量快照与问题表

- `data_quality_snapshot`
- `data_quality_issue`

设计作用：

- 以市场维度保存最近一次可解释的数据质量快照，而不是把质量判断散落在前端或临时脚本里
- 将问题明细标准化入库，便于 `/api/data-quality/issues` 直接查询和筛选
- 为后续补齐真实采集器、调度任务和质量中心看板预留统一存储契约

### 5.7 关键字段级说明

#### `trading_price_YYYY`

核心字段：

- `settlement_date`：结算时间
- `region_id`：区域或市场标识，如 `NSW1`、`QLD1`、`WEM`
- `rrp_aud_mwh`：能量价格

NEM 常见 FCAS 字段：

- `raise1sec_rrp`
- `raise6sec_rrp`
- `raise60sec_rrp`
- `raise5min_rrp`
- `raisereg_rrp`
- `lower1sec_rrp`
- `lower6sec_rrp`
- `lower60sec_rrp`
- `lower5min_rrp`
- `lowerreg_rrp`

业务含义：

- 这个表是澳洲主工作台的时序底座
- 价格趋势、峰谷套利、小时剖面、NEM FCAS 分析都依赖它

#### `wem_ess_market_price`

核心字段分组：

- 价格字段：
  - `energy_price`
  - `regulation_raise_price`
  - `regulation_lower_price`
  - `contingency_raise_price`
  - `contingency_lower_price`
  - `rocof_price`
- 可用量字段：
  - `available_regulation_raise`
  - `available_regulation_lower`
  - `available_contingency_raise`
  - `available_contingency_lower`
  - `available_rocof`
- 在服量字段：
  - `in_service_regulation_raise`
  - `in_service_regulation_lower`
  - `in_service_contingency_raise`
  - `in_service_contingency_lower`
  - `in_service_rocof`
- 需求字段：
  - `requirement_regulation_raise`
  - `requirement_regulation_lower`
  - `requirement_contingency_raise`
  - `requirement_contingency_lower`
  - `requirement_rocof`
- 缺口字段：
  - `shortfall_regulation_raise`
  - `shortfall_regulation_lower`
  - `shortfall_contingency_raise`
  - `shortfall_contingency_lower`
  - `shortfall_rocof`
- 出清量字段：
  - `dispatch_total_regulation_raise`
  - `dispatch_total_regulation_lower`
  - `dispatch_total_contingency_raise`
  - `dispatch_total_contingency_lower`
  - `dispatch_total_rocof`
- 封顶标记：
  - `capped_regulation_raise`
  - `capped_regulation_lower`
  - `capped_contingency_raise`
  - `capped_contingency_lower`
  - `capped_rocof`

业务含义：

- 这是 WEM FCAS preview 与 WEM 预测的关键数据表
- 它不是完整市场仓库，而是分析导向的 slim 表

#### `wem_ess_constraint_summary`

核心字段：

- `binding_count`
- `near_binding_count`
- `binding_max_shadow_price`
- `near_binding_max_shadow_price`
- `max_formulation_shadow_price`
- `max_facility_shadow_price`
- `max_network_shadow_price`
- `max_generic_shadow_price`

业务含义：

- 反映约束紧张程度
- 主要服务于事件解释和 WEM 预测信号

#### `wem_ess_capability`

核心字段：

- `facility_code`
- `participant_code`
- `participant_name`
- `facility_class`
- 各服务 `max_accredited_*`
- `facility_speed_factor`
- `rocof_ride_through_capability`
- `extracted_at`

业务含义：

- 描述参与者或设施的能力边界
- 更偏静态元数据，不是主时序分析底座

#### `grid_event_raw` 与 `grid_event_state`

`grid_event_raw` 保存原始事件事实，`grid_event_state` 保存归一化后的状态判断。  
前者更像证据库，后者更像前端与预测引擎可直接消费的解释层。

#### `fingrid_dataset_catalog` / `fingrid_timeseries` / `fingrid_sync_state`

- `dataset_catalog`：定义支持的数据集、单位、启用状态和来源
- `timeseries`：保存标准化后的时序值
- `sync_state`：保存上次同步时间、错误、窗口和覆盖度

这三张表共同构成 Fingrid 子系统的最小闭环。

## 6.4 接口风格

统一特征：

- 以 JSON 为主
- 大部分接口围绕前端图表消费设计
- 兼顾业务摘要字段与原始时序字段
- 关键分析接口开始统一返回 `metadata`，用于表达 `freshness`、`source_version`、`timezone`、`interval_minutes`、`quality_grade` 与 `coverage`
- 重点接口带缓存
- 错误响应多为 HTTPException 包装后的 4xx / 5xx

### 6.7 主接口处理链路

这一节不是重复接口定义，而是说明“一个请求进来之后，系统实际按什么顺序处理”。

#### `/api/price-trend` 请求生命周期

1. 前端带上 `year`、`region`、可选 `month/quarter/weekend_only/workday_only/limit` 发起请求。
2. 后端根据 `year` 定位到 `trading_price_YYYY` 表。
3. 组装 SQL 条件，按区域和时间切片过滤价格序列。
4. 从数据库取出时序行后，计算基础统计值：
   - 最小值
   - 最大值
   - 平均值
   - 负价占比
   - 样本数
5. 如果点数超过 `limit`，优先走 `lttbc` 做 LTTB 降采样。
6. 如果 `lttbc` 或 `numpy` 不可用、运行报错，自动降级到均匀抽样，不阻断接口返回。
7. 返回图表序列和顶部 KPI。

这条链路说明一个重要事实：`price-trend` 的核心不是“复杂模型”，而是“数据库过滤 + 统计摘要 + 可视化友好的采样”。

#### `/api/peak-analysis` 请求生命周期

1. 接收市场、区域、年份、聚合粒度以及网络费参数。
2. 从年度价格表提取原始序列。
3. 按 1h / 2h / 4h / 6h 等窗口重组价格。
4. 计算每个窗口的：
   - 充电均价
   - 放电均价
   - 毛价差
   - 扣除网络费后的净价差
5. 生成分布、排序、阈值判断和代表性窗口。
6. 结果同时被 `PeakAnalysis`、`RevenueStacking`、`CycleCost`、`BessSimulator` 等前端模块复用。

#### `/api/fcas-analysis` 请求生命周期

1. 根据市场分流到 NEM 或 WEM 分支。
2. NEM 分支直接读取年度价格表中的 FCAS 字段。
3. WEM 分支读取 `wem_ess_market_price` 为主，必要时联动约束/能力信息补充解释。
4. 生成服务拆分、小时分布、趋势线和汇总统计。
5. 返回给 `FcasAnalysis` 和 `RevenueStacking`。

#### `/api/grid-forecast` 请求生命周期

1. 接收 `market`、`region`、`horizon`、`as_of`。
2. 先尝试从 `grid_forecast_snapshot` 读取对应快照。
3. 若无合适缓存，则从事件状态、近端价格、WEM slim / NEM 历史等上游源重建预测输入。
4. 进入 `grid_forecast.py` 的规则引擎。
5. 产出：
   - `summary`
   - `windows`
   - `drivers`
   - `coverage/confidence` 类元数据
6. 结果写回快照表，供后续相同请求复用。

#### `/api/investment-analysis` 请求生命周期

1. 前端提交完整 `InvestmentParams`。
2. 后端先做 Pydantic 参数校验。
3. 生成分析请求键，检查 `analysis_cache` 中是否已有相同输入结果。
4. 若命中缓存，直接返回，避免重复重算。
5. 若未命中：
   - 读取套利和 FCAS 相关基础收益口径
   - 进入 `backend/engines` 财务模型链
   - 计算现金流、回收期、IRR、ROI、NPV、敏感性和蒙特卡洛结果
6. 将结果写入 `analysis_cache`。
7. 返回给前端投资分析模块。

#### Fingrid `/series` 与 `/summary` 请求生命周期

1. 路由先校验数据集是否存在于 catalog。
2. 由 `fingrid_service` 统一处理时区、聚合粒度、时间窗口。
3. 从 `fingrid_timeseries` 拉取原始行。
4. `series` 路径负责原始或聚合时序输出。
5. `summary` 路径在此基础上继续计算均值、峰值、谷值、分布等摘要。
6. 前端再决定卡片、折线、分布面板怎么展示。

## 8. 核心算法与模型逐层解析

## 8.1 价格趋势

业务理解：

- 用于看价格怎么走、极值在哪里、负价有多常见

工程实现：

- 后端从 `trading_price_YYYY` 取时序
- 在点数过多时做下采样
- 同时给出最小值、最大值、均值、负价比例等统计

## 8.2 峰谷套利分析

业务理解：

- 不是“真实调度执行收益”，而是寻找理论上的价差机会窗口

工程实现：

- 基于不同结算粒度生成 1h / 2h / 4h / 6h 窗口
- 计算 gross / net spread
- WEM 和 NEM 因结算粒度不同而走不同窗口点数

## 8.3 FCAS 分析

业务理解：

- NEM 更接近历史价格分析
- WEM 更接近基于 slim 表的 preview 估算

工程实现：

- NEM 依赖主价格表中的 FCAS 列
- WEM 依赖独立 WEM ESS slim 表
- 二者在实现和可信边界上不是对称关系

## 8.4 事件层

业务理解：

- 不是单纯列出新闻，而是把事件归一化成对分析更有意义的状态标签

工程实现：

- `grid_events.py` 负责事件归类、影响域映射和时间过滤
- 下游既服务图表解释，也服务预测引擎

## 8.5 电网预测

业务理解：

- 是规则型信号引擎，不是 ML 模型
- 输出的是风险和机会信号，而不是价格精确预测

工程实现：

- `grid_forecast.py` 负责 NEM / WEM 分支
- NEM 与 WEM 使用不同输入与输出可信度
- `grid-forecast/coverage` 单独输出覆盖质量摘要

## 8.6 投资分析

业务理解：

- 这是一个“带明确假设的财务情景模型”
- 不等于资产历史真实收益复盘

工程实现：

- `POST /api/investment-analysis`
- 参数模型位于 `backend/models/financial_params.py`
- 财务计算位于 `backend/engines/financial_model.py`
- 同时结合套利、FCAS、折现、衰减、债务容量、杠杆 IRR 等要素

## 8.7 BESS 回测与辅助引擎

相关模块：

- `bess_backtest.py`
- `engines/battery_model.py`
- `engines/dispatch_optimizer.py`
- `engines/revenue_model.py`

它们共同承担：

- 储能物理约束建模
- 调度 / 收益估算
- 财务输入生成

### 8.8 引擎层职责拆分

#### `battery_model.py`

- 负责电池衰减、寿命、增强策略等物理状态演化

#### `dispatch_optimizer.py`

- 负责调度优化逻辑
- 更偏“给定价格路径下的最优行为近似”

#### `revenue_model.py`

- 负责收益侧估算，尤其是套利与 FCAS 的组合口径

#### `financial_model.py`

- 负责 NPV、IRR、ROI、Payback、Debt Capacity、Levered IRR 等财务计算

#### `market_adapter.py`

- 负责把市场数据查询抽象成分析可消费的接口

它们的存在说明投资分析并不是一坨写死在 `server.py` 里的公式，而是已经拆成多个可演进的责任单元。

---

## 9. 业务口径、结果解释与适用边界

### 9.1 读结果时必须记住的事情

- 价格分析不等于收益分析
- 理论套利机会不等于真实可执行收益
- WEM 当前很多能力是 preview，不是完整投资级结论
- `metadata.quality_grade` 如果是 `preview` 或 `analytical-preview`，应理解为“可用于方向分析”，而不是“已完成严谨数据审计”
- 投资分析是模型结果，不是历史已实现现金流
- 预测模块是规则信号，不是 ML 点预测

### 9.2 NEM 与 WEM 的不对称性

虽然前端都能展示为“市场模块”，但当前事实是：

- NEM 的历史口径更完整
- WEM 的部分能力依赖 slim 表
- WEM 在 FCAS、约束和预测层面更适合方向判断，而不是精算结论

### 9.3 Fingrid 的边界

Fingrid 当前更像独立数据分析页，不和澳洲主工作台共享一套业务模型。  
它已形成自己的采集、聚合、摘要和导出路径，但尚未和 BESS 财务模型深度耦合。

## 10. 缓存、调度、同步与运行机制

## 10.1 响应缓存

当前使用：

涉及范围：

- `peak-analysis`
- `fcas-analysis`
- `hourly-price-profile`
- `event-overlays`
- `grid-forecast`
- `investment-analysis`

作用：

- 减少重复计算
- 降低前端频繁切换参数时的响应压力

## 10.2 分析缓存

SQLite 中的 `analysis_cache` 用于更重的分析结果复用，尤其是投资分析相关结果。

## 10.3 内置调度器

后端通过 APScheduler 提供内置调度能力。

当前调度方向：

- 市场日常同步
- Fingrid 小时级增量同步

可以通过环境变量禁用：

- `AUS_ELE_ENABLE_SCHEDULER=false`

## 10.4 手动同步

市场主链：

- `POST /api/sync_data`

Fingrid：

- `POST /api/fingrid/datasets/{dataset_id}/sync`

这说明两条数据线在同步机制上已经分开设计。

---

## 10.5 同步任务逐过程拆解

### `run_sync_scrapers`

这是澳洲市场主同步任务，顺序非常明确：

1. 获取市场同步锁，避免重复任务重叠执行。
2. 计算最近两周时间窗。
3. 执行 `aemo_wem_scraper.py`，补近 14 天 WEM 价格。
4. 执行 `aemo_wem_ess_scraper.py`，补近 30 天 WEM ESS slim 数据。
5. 执行 `aemo_nem_scraper.py`，补近两周所在月份的 NEM 数据，并带 FCAS。
6. 执行 `aemo_grid_event_scraper.py`，补近 180 天事件。
7. 写入 `last_update_time`。
8. 释放锁。

从这个顺序可以看出，它是一个“多脚本编排器”，不是一个单 SQL/单模块同步器。

### `run_fingrid_dataset_sync`

这是 Fingrid 单数据集同步入口：

1. 先检查 `FINGRID_API_KEY` 是否配置。
2. 获取 Fingrid 同步锁。
3. 调 `fingrid_service.sync_dataset(...)`。
4. 成功时返回 `dataset_id / mode / records_upserted / last_synced_timestamp_utc`。
5. 失败时返回结构化错误对象，而不是只打日志。
6. 最终释放锁。

### `fingrid_service.sync_dataset`

这是 Fingrid 同步真正的核心过程：

1. 从 catalog 读取数据集配置。
2. 如果是 `backfill`，使用配置中的默认起始时间；如果是 `incremental`，使用默认回看天数。
3. 将时间窗切成按月窗口，避免一次拉太大范围。
4. 每个窗口调用 Fingrid API。
5. 每条原始行经过 `normalize_fingrid_row` 标准化。
6. 写入 `fingrid_timeseries`。
7. 更新 `fingrid_sync_state` 为 `running`、`ok` 或 `error`。

这说明 Fingrid 不是“抓完直接展示”，而是“先标准化入库，再以统一查询接口提供数据服务”。

### `run_fingrid_hourly_sync`

1. 先检查 API Key。
2. 获取全局 Fingrid 同步锁。
3. 遍历 catalog 中所有支持的数据集。
4. 对每个数据集跑一次 `incremental` 同步。
5. 汇总成功数、失败数和失败明细。
6. 返回结构化结果供日志或状态面板消费。

## 10.6 锁、缓存与并发边界

### 锁机制

- 市场同步和 Fingrid 同步使用不同的 job lock 名称。
- 锁的目标不是“提升性能”，而是防止重复抓取、重复写库和状态污染。
- 手动触发接口在拿不到锁时会直接返回冲突或 `skipped`，而不是排队无限等待。

### Redis 响应缓存

- `response_cache.py` 封装了 `RedisResponseCache`。
- Redis 不可达、依赖未安装、超时等情况不会让接口整体失败。
- 失败时行为是：
  - 记录 warning
  - 将内部 client 清空
  - 本次请求退化为无缓存直算

### SQLite 并发边界

- 当前项目是 SQLite 架构，适合单机部署、轻中度并发和分析型查询。
- 它不适合高频写入和多实例共享写竞争。
- 同步任务、快照写入、分析缓存写入都在争用同一个本地数据库文件时，要关注锁等待和 I/O 抖动。

## 13. 当前限制、风险点与非目标

### 13.1 文档层面

- 历史分析、审计、需求和现状文档并存，容易互相冲突
- 旧文档里仍有过时接口名、表名和历史结论

### 13.2 数据与算法层面

- WEM 与 NEM 的能力并不对称
- WEM 的部分结论仍属于 preview 层
- 数据质量中心 v1 已经补上了统一快照、问题表和最小 collector，当前可直接从 NEM 年度价格表、WEM ESS slim 表、Fingrid 同步状态与时序表生成质量快照
- `price-trend` 与 Fingrid 状态虽然已经统一输出 `metadata`，但这不代表所有历史接口都完成了同样粒度的结果治理
- 投资分析依赖模型假设，不是已实现收益
- 预测是规则引擎，不是机器学习价格预测

### 13.3 工程层面

- SQLite 适合单机和轻量部署，但不适合高并发多写场景
- Redis 不可用时系统会退化为无缓存路径
- 内置调度器在多副本部署时需要显式控制

### 13.4 产品层面

- 澳洲主工作台与 Fingrid 页面目前仍是两条产品线
- 还没有统一成跨市场、跨区域、跨收入口径的单一分析框架

### 13.5 运行时故障模式与排障入口

#### `price-trend` 500 或图表空白

优先排查：

1. 对应年份的 `trading_price_YYYY` 是否存在。
2. 目标 `region` 在该年份表中是否有数据。
3. 是否传入了过严的月份/季度/工作日过滤导致结果集为空。
4. 若点数很多，观察日志里是否出现 LTTB 降级 warning。

注意：LTTB 降级本身通常不会导致接口失败，它更多影响的是采样质量，不是可用性。

#### Fingrid 页面无数据

优先排查：

1. `FINGRID_API_KEY` 是否配置。
2. `fingrid_sync_state` 是否长期停留在 `error`。
3. `fingrid_timeseries` 是否已有该 `dataset_id` 数据。
4. 请求时间窗是否超出已同步覆盖范围。

#### 手动同步无响应或返回冲突

优先排查：

1. 是否已有后台同步任务在运行。
2. 系统锁是否未正常释放。
3. SQLite 是否被长事务占用。

#### 投资分析过慢

优先排查：

1. 相同参数是否命中 `analysis_cache`。
2. 是否开启了敏感性 / 蒙特卡洛并设置了较大迭代数。
3. 是否在资源较小的容器中运行导致 CPU 计算变慢。

#### Redis 不可用

预期行为：

- 不应造成整体服务启动失败。
- 接口会退化为无响应缓存模式。
- 影响主要体现在重复请求延迟上升，而不是功能缺失。
