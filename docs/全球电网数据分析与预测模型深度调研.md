# 全球电网数据分析与预测模型深度调研

**日期**: 2026-04-30  
**范围**: 当前项目真实算法现状 + 全球主要电力市场研究入口 + 预测/分析模型升级路线  
**重点区域**: 澳洲、欧洲、美国、日本、东南亚

---

## 1. 结论先行

当前项目已经具备一套**不错的电力市场分析产品雏形**，但从“真正的预测平台”标准看，核心能力还处在：

1. **历史统计分析**: 已经比较完整。
2. **规则型信号预测**: 已经成型，但本质还是 heuristic engine，不是严格意义上的 forecasting model。
3. **储能理论收益上界回测**: 已经有，但主要是 hindsight upper bound，不是可执行的实盘策略回测。
4. **概率预测、基本面预测、状态识别、联合优化**: 基本还没有建立成体系。

所以，当前系统更准确的定位不是“预测引擎平台”，而是：

> **以电价、负价、FCAS/ESS 机会识别为核心的研究型分析工作台 + 轻量级规则信号前瞻层。**

如果你下一阶段要把它做成更强的商业级产品，真正缺的不是单点“更准的 price forecast”，而是整套能力栈：

- **基本面数据层**: load / wind / solar / outage / transmission / interconnector / reserve / weather
- **状态建模层**: scarcity regime / oversupply regime / congestion regime / reserve tightness regime
- **概率预测层**: P10 / P50 / P90，而不是单点分数
- **决策优化层**: stochastic dispatch / rolling horizon / ancillary stacking
- **评估治理层**: walk-forward backtest / forecast calibration / error attribution / data lineage

## 2. 当前项目“真实模型”到底是什么

### 2.1 `grid_forecast.py` 不是 ML 预测模型，而是规则型信号引擎

从代码实现看，当前 forecast 主链路主要在：

- [backend/grid_forecast.py](/g:/project/aus-ele/backend/grid_forecast.py:227)
- [backend/grid_forecast.py](/g:/project/aus-ele/backend/grid_forecast.py:420)
- [backend/grid_forecast.py](/g:/project/aus-ele/backend/grid_forecast.py:582)
- [backend/grid_forecast.py](/g:/project/aus-ele/backend/grid_forecast.py:765)

它现在做的事情本质是：

1. 取最近一段历史价格/FCAS/约束/shortfall 数据；
2. 拼接 event overlay / predispatch / slim WEM tables；
3. 用固定公式打分：
   - `grid_stress_score`
   - `price_spike_risk_score`
   - `negative_price_risk_score`
   - `reserve_tightness_risk_score`
   - `fcas_opportunity_score`
   - `charge_window_score`
   - `discharge_window_score`
4. 输出 windows / drivers / warnings / confidence band

几个关键事实：

- NEM 24h forecast 标记为 `forecast_mode = "hybrid_signal_calibrated"`，但本质仍是规则融合，不是训练得到的 statistical model。见 [backend/grid_forecast.py](/g:/project/aus-ele/backend/grid_forecast.py:552)
- 7d / 30d 输出是 regime outlook，不是逐时价格预测。见 [backend/grid_forecast.py](/g:/project/aus-ele/backend/grid_forecast.py:678)
- WEM forecast 仍是 `core_only` / preview-grade。见 [backend/grid_forecast.py](/g:/project/aus-ele/backend/grid_forecast.py:851)
- 所有 forecast 返回里都明确写了 `investment_grade = False`。见：
  - [backend/grid_forecast.py](/g:/project/aus-ele/backend/grid_forecast.py:558)
  - [backend/grid_forecast.py](/g:/project/aus-ele/backend/grid_forecast.py:717)
  - [backend/grid_forecast.py](/g:/project/aus-ele/backend/grid_forecast.py:857)

**判断**:

这是一套“可解释、可产品化、响应快”的前瞻信号系统，但不是严格意义上的价格预测模型，更不是资产调度级预测模型。

### 2.2 `bess_backtest.py` 是 hindsight upper bound，不是实盘策略回测

主实现见：

- [backend/bess_backtest.py](/g:/project/aus-ele/backend/bess_backtest.py:99)

核心特征：

- 用 MILP 做优化；
- 月度内固定 50% 初始 SoC；
- 月末强制回到 50% SoC。见 [backend/bess_backtest.py](/g:/project/aus-ele/backend/bess_backtest.py:173)
- 返回结果明确标记：
  - `backtest_mode = "optimized_hindsight"`
  - `revenue_scope = "physical_upper_bound"`
  见 [backend/bess_backtest.py](/g:/project/aus-ele/backend/bess_backtest.py:211)

**判断**:

它适合回答的问题是：

- “这个市场在历史上对某类储能时长有没有套利空间？”
- “理论上限大概在哪？”

它不适合直接回答：

- “这个 BESS 在未来一年真实可拿到多少收益？”
- “如果接入 rolling forecast / FCAS stack / SoC risk control，实盘表现如何？”

### 2.3 投资分析已经比旧版本规范，但仍然依赖 upper-bound 基线

相关位置：

- [backend/models/financial_params.py](/g:/project/aus-ele/backend/models/financial_params.py:57)

这里有两个重要结论：

1. 之前“`degradation_rate` 没生效”的问题现在已经不是现状了。

2. 当前投资模型仍然是：
   - 先拿 standardized backtest 结果做基线；
   - 再用 `forecast_inefficiency` 做 haircut。见：
     - [backend/models/financial_params.py](/g:/project/aus-ele/backend/models/financial_params.py:74)

**判断**:

这比早期版本已经合理得多，但依然不是“forecast-driven dispatch valuation”。  
它更像：

> **基于历史标准化回测的收益基线 + 经验折扣的投资测算。**

### 2.4 WEM FCAS/ESS 仍然偏 preview / heuristic

WEM ESS 分析主实现见：

- [backend/server.py](/g:/project/aus-ele/backend/server.py:3961)

这里仍有一个值得单独记住的问题：

- `_get_wem_ess_analysis()` 里仍然写死 `interval_hours = 5 / 60`。见 [backend/server.py](/g:/project/aus-ele/backend/server.py:3971)

这意味着：

- 如果未来 WEM 数据源粒度、回灌逻辑或多类型时间粒度发生变化，这里会先变成误差源；
- 相比 NEM standardized backtest 路径中“根据时间戳推断 interval”的做法，WEM 这段仍偏简化。

## 3. 当前系统真正擅长什么

如果客观评价，当前系统真正强的不是“精确预测”，而是下面四件事：

### 3.1 市场形态识别

它很擅长识别：

- 高波动区域
- 负价窗口
- 日内 spread 结构
- FCAS/ESS scarcity 信号
- 特定年份/月份/工作日类型的风险特征

### 3.2 解释层

你现在的产品比很多纯模型 demo 更强的一点，是它已经有了：

- event overlay
- driver tags
- structured metadata
- coverage / freshness / warning

这会让结果更可解释、更适合商业化。

### 3.3 筛选和机会发现

它已经适合做：

- 哪个 market / region 更值得研究
- 哪个年份/月份有异常价差
- 哪些时段更适合 charge / discharge / ancillary

### 3.4 原型级资产评估

虽然不是项目融资级模型，但已经足够做：

- 市场对比
- duration 方向筛选
- early-stage feasibility

## 4. 当前系统真正缺什么

### 4.1 缺少基本面预测，而不是只缺价格预测

真正的电价预测通常不是直接拿 price 去做单变量时间序列。

更稳的做法是先建这些基础预测：

- load / demand
- wind forecast
- solar forecast
- rooftop PV
- outage / unit availability
- reserve requirement
- transmission congestion / interconnector flow
- weather / temperature / irradiance / wind speed

然后再进入 price / ancillary / scarcity 模型。

### 4.2 缺少 regime / state model

现在系统会输出 “risk score”，但没有一个显式的状态空间，例如：

- normal regime
- solar oversupply regime
- evening scarcity regime
- thermal outage regime
- FCAS tightness regime
- transmission separation regime

没有 regime layer，会带来两个问题：

1. 模型解释性还不够结构化；
2. 预测和优化无法针对不同状态切换策略。

### 4.3 缺少概率预测

当前输出更像 score，不像 distribution。

商业级预测最少要做到：

- P10 / P50 / P90
- quantile band
- confidence calibration
- interval coverage tracking

因为储能和风险管理最怕的不是点预测误差，而是**尾部场景没有被建模**。

### 4.4 缺少 decision-focused optimization

真正有价值的不是“价格预测更准一点”，而是：

> 用预测结果驱动更好的 dispatch / bidding / reserve participation 决策。

当前还缺：

- rolling horizon optimization
- stochastic programming
- ancillary + energy co-optimization
- SoC risk reserve
- degradation-aware dispatch
- imbalance / settlement exposure integration

### 4.5 缺少系统化评估闭环

真正的模型平台必须长期回答这些问题：

- 上周预测误差在哪些 regime 爆了？
- 哪类输入缺失最影响结果？
- price model 还是 renewable forecast 在拖后腿？
- 预测更准是否真的提高 dispatch revenue？

当前项目这层还没有成体系。

## 5. 你应该如何理解“真正的预测平台”

建议把目标拆成 6 层：

### 5.1 数据层

采集并治理：

- 实时/历史价格
- dispatch
- reserve / FCAS / ancillary
- demand/load
- weather
- outages
- interconnector / flow
- network constraints
- renewable forecast / actual
- settlement / imbalance / curtailment

### 5.2 特征层

构建：

- 日内位置特征
- ramp / volatility / spread / kurtosis
- renewable penetration
- thermal outage intensity
- reserve tightness
- price separation / congestion
- weather anomalies
- cross-market lead-lag

### 5.3 基本面层

先预测：

- demand
- wind
- solar
- reserve requirement
- outage availability

### 5.4 市场状态层

识别 regime：

- oversupply
- scarcity
- congestion
- ancillary stress
- structural transition

### 5.5 价格与服务预测层

输出：

- spot price quantiles
- FCAS / ESS service price quantiles
- spike probability
- negative-price duration probability

### 5.6 资产决策层

对 BESS / flexible load / hybrid asset 做：

- rolling dispatch
- co-optimization
- risk budgeting
- revenue attribution
- value decomposition

## 7. 研究模型时，应该优先学什么

### 7.1 官方规则与方法论

先看市场运营者和监管方，因为这决定：

- 价格如何形成
- reserve / ancillary 如何清算
- forecast 数据在什么时间发布
- imbalance / settlement 怎么影响真实收益

### 7.2 系统规划与供需预测方法

必须看：

- demand forecast methodology
- adequacy assessment
- transmission / interconnector planning
- renewable integration studies

因为很多 spike / negative / curtailment 都不是纯统计现象，而是结构性现象。

### 7.3 交易所与市场数据接口

必须搞清楚：

- day-ahead
- intraday
- balancing
- reserve
- historical vs real-time vs delayed

### 7.4 学术模型，但要筛选

不要被论文数量带偏。真正值得优先看的是：

- probabilistic electricity price forecasting
- regime-switching / state-space models
- weather-to-price feature pipelines
- stochastic storage optimization
- forecast value / decision-focused evaluation

可作为研究入口的论文与综述：

- Probabilistic multivariate electricity price forecasting using implicit generative ensemble post-processing  
  https://arxiv.org/abs/2005.13417

## 8. 统一升级路线：P0 到 P4

结合当前代码现状、外部市场调研结果，以及你提出的分层，我建议把后续升级统一收敛成下面这条主路线。

### 8.1 P0：补数据底座

目标不是“多接几个表”，而是建立能支撑 forecast / regime / dispatch / valuation 的基础观测层。

核心范围：

- load forecast / actual
- wind forecast / actual
- solar forecast / actual
- rooftop PV
- outage / unit availability
- interconnector flow
- reserve / shortfall
- weather
- constraint
- settlement

这里需要再拆清楚两层：

1. **原始输入层**
   - forecast / actual
   - constraint equation / binding flag / shadow price
   - interconnector flow / limit / outage
   - reserve requirement / shortfall
   - settlement / imbalance / realized revenue inputs
2. **统一建模层**
   - 时间对齐
   - 区域映射
   - 市场映射
   - coverage / freshness / quality metadata

这一层是整个路线的前提。没有 P0，P1-P4 都会变成不稳定的上层拼装。

### 8.2 P1：把当前 score 升级为显式 regime layer

当前系统已经有一套不错的 score 和 driver 结构，但还缺显式状态层。

建议 regime 统一定义为：

- oversupply
- scarcity
- negative price
- reserve stress
- congestion
- transmission separation

这里需要特别强调一件事：

- `constraint` 与 `interconnector flow` 属于 **P0 输入**
- `congestion` 与 `transmission separation` 属于 **P1 解释状态**

也就是说，constraint / flow 是原料，regime 是结论。

这一层建起来后，现有：

- `grid_forecast`
- `event_overlays`
- `market_screening`
- `fcas_analysis`
- `investment_analysis`

都应该逐步从“散落的 score”转向“score + regime + explanation”的统一表达。

### 8.3 P2：做可评估预测层

这一层的目标不是追求最复杂模型，而是先建立**可信的预测生产线**。

建议顺序：

1. baseline point forecast
2. quantile forecast
3. spike probability
4. negative price duration

但这里必须再补一层 benchmark hierarchy：

- naive baseline
- seasonal baseline
- fundamental baseline
- ML baseline
- probabilistic model

否则会出现“模型更复杂了，但没有实质超越简单基线”的假进步。

同时，这一层必须自带评估机制：

- walk-forward backtest
- calibration
- regime error attribution

如果没有这三项，forecast 只能算结果展示，不能算生产级能力。

### 8.4 P3：做 BESS 决策层

当前 `bess_backtest.py` 更像 hindsight upper bound。P3 的目标是把它推进成真正的决策层。

核心能力：

- rolling horizon dispatch
- energy + FCAS/ESS co-optimization
- degradation-aware dispatch
- scenario / stochastic dispatch
- revenue attribution

这一层真正要回答的问题是：

- 预测提升后，dispatch 是否变好？
- multi-service stacking 是否提升收益质量？
- uncertainty 是否改变最优 SoC 策略？
- degradation / risk reserve 是否改变长期价值？

### 8.5 P4：做模型治理层

P4 不是“收尾项”，而是可信商业化能力。

核心范围：

- data lineage
- source freshness
- drift detection
- forecast value attribution
- investment-grade disclaimer

其中 `investment-grade disclaimer` 不应该等到 P4 完成才出现，而应该从现在开始贯穿全链路；P4 做的是把它系统化、标准化、可审计化。

## 9. 这条路线和原始调研结论的对应关系

为了避免后续执行时口径漂移，可以把 P0-P4 和前文调研结论做一一对应：

- **P0 数据底座** 对应前文“缺基本面数据体系”
- **P1 regime layer** 对应前文“缺显式状态空间”
- **P2 可评估预测层** 对应前文“缺概率预测与评估闭环”
- **P3 BESS 决策层** 对应前文“缺 forecast-to-decision 优化”
- **P4 治理层** 对应前文“缺 lineage / freshness / disclaimer / drift / attribution”

这样后面无论是拆 spec、写计划，还是做阶段验收，都有同一套顶层框架。

## 10. 对这个仓库最现实的执行顺序

如果以当前仓库结构和现有能力为基线，最现实的顺序是：

### Phase A：先把 P0 做出最小可用底座

优先接入最有价值、最能驱动后续建模的数据：

- load forecast / actual
- wind / solar forecast / actual
- rooftop PV
- outage
- interconnector flow
- reserve / shortfall
- weather
- constraint

`settlement` 可先进入 schema 与 lineage 设计，再逐步进入收益兑现链路。

### Phase B：用 P1 重构现有分析表达

先不推翻现有 score 系统，而是：

- 保留原有 score
- 新增统一 regime layer
- 把现有 drivers / summaries / overlays 全部挂到 regime 上

这是对现有产品最平滑、最可控的增强方式。

### Phase C：在 P1 之上建立 P2 预测层

顺序建议：

1. fundamentals baseline
2. point forecast
3. quantile forecast
4. spike / negative-price specialized models
5. walk-forward + calibration + regime attribution

### Phase D：把 forecast 接到 P3 决策层

不是“先做一个更漂亮的 backtest 页面”，而是把：

- forecast
- regime
- asset constraints
- FCAS/ESS stack

真正接成 dispatch 与 revenue engine。

### Phase E：用 P4 收拢治理

最终形成统一的：

- source-level lineage
- freshness / quality
- drift / calibration status
- applicability statement
- investment-grade disclaimer
