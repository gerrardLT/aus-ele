## 2. 执行摘要

### 2.1 总体判断

当前系统适合做:

- 市场价格观察
- 负电价时段识别
- 粗粒度套利机会展示
- 粗略投资情景演示

当前系统不适合直接做:

- WEM/NEM 多市场联合收益判断
- FCAS/ESS 真实收益测算
- 生产级储能调度策略评估
- 可直接用于投委会或融资材料的投资结论输出

### 2.2 核心原因

1. 收益建模口径存在混淆，价格与收入被部分直接混用。
2. WEM ESS 抓取链路代码存在，但未接入实际同步任务。
3. 前端暴露了超出当前数据与算法能力边界的分析入口。
4. 套利与投资模块仍存在完美预见或简化假设，容易高估收益。

## 3. 关键问题清单

以下按严重度排序。

### [P0] 收入叠加图将“价格”直接当作“收入”相加

**位置**

- `web/src/components/RevenueStacking.jsx:48-80`
- `server.py:579-666`

**已验证事实**

- `RevenueStacking` 将 `peak-analysis` 返回的 `net_spread_4h/spread_4h` 与各 FCAS 价格列直接合并成同一条 stacked area。
- `fcas-analysis` 返回的是各服务的平均价格及简化估算收入，但时间序列部分仍是价格均值。
- 当前前端展示会让用户直观认为“套利收入 + FCAS 收入”已在同一口径下叠加。

**影响**

- 量纲错误。
- 用户会把价格序列误读为收益序列。
- 会直接误导“收入构成”“市场机会大小”和“储能配置优先级”判断。

**正确口径**

收入至少应满足:

`收入 = 价格 × 启用/可用容量 × 结算时长 × 可用率/启用率`

如果存在并行服务，还必须额外受限于:

- 充放电方向
- SoC
- 功率上限
- headroom / footroom
- 同时服务占容

**建议**

- 彻底拆分“价格分析”和“收益分析”两个模块。
- 收益模块应改为容量受限、可用率显式化的收益估算或优化模型。

### [P1] 投资模型中的 `degradation_rate` 输入参数未真正生效

**位置**

- `server.py:697-718`
- `server.py:778-803`

**已验证事实**

- `InvestmentParams` 定义了用户输入项 `degradation_rate`。
- 实际现金流计算未使用该输入值。
- 计算中使用的是硬编码的:
  - `calendar_deg_rate = 0.015`
  - `cycle_deg_rate = annual_full_cycles * (0.20 / 6000)`
  - `dynamic_degradation_rate = calendar_deg_rate + cycle_deg_rate`

**影响**

- 前端参数面板存在“可调但不生效”的假交互。
- 用户会误以为自己已经完成退化敏感性分析。
- 投资模型的可解释性和可信度明显下降。

**建议**

- 明确选择一种口径:
  - 用户输入退化率并直接生效
  - 或系统使用双因子退化模型并在前端隐藏单独退化率参数
- 如果保留双因子模型，应将公式与关键参数显式展示给用户。

**位置**

- `server.py:60-79`
- `server.py:83-87`

**已验证事实**

- 自动同步与手动同步只执行:
  - `aemo_wem_scraper.py`

- 没有任何地方调用 `aemo_wem_ess_scraper.py`

**影响**

- 前端对 WEM 的支持边界与真实数据能力不一致。
- 用户容易误解为“WEM 没有辅助服务数据”而不是“系统没同步”。

**建议**

- 把 `aemo_wem_ess_scraper.py` 正式纳入定时同步和手动同步。
- 增加状态页或摘要指标，明确显示 `wem_ess_price` 数据覆盖范围。

**位置**

- `web/src/App.jsx:309`

  - NEM FCAS
  - WEM ESS
- 在 WEM 下展示单独状态:
  - “WEM ESS 尚未同步”
  - “仅支持 RTP，不支持 ESS 收益”

### [P1] 套利分析和循环成本分析仍是“机会值”逻辑，不是“可执行收益”逻辑

**位置**

- `server.py:273-375`
- `web/src/components/CycleCost.jsx:35-74`

**已验证事实**

- `peak-analysis` 采用逐日滑窗扫描，寻找最佳 peak/trough 窗口。
- 该逻辑没有显式建模:
  - SoC 状态连续性
  - 充放电互斥
  - 期末 SoC 目标
  - 并行服务占容
- `CycleCost` 用的是 `spread_4h`，而不是 `net_spread_4h`

**影响**

- 结果更像“理论机会值”而不是“可交付收益”。
- `CycleCost` 容易高估“值得循环的天数”。

**建议**

- 把 `PeakAnalysis` 明确标注为“理论机会分析”。
- `CycleCost` 至少切换到 `net_spread_4h`。
- 中长期应统一迁移到约束优化或滚动调度模型。

### [P2] 1 秒 FCAS 已在代码层半接入，但数据库与前后端仍主要停留在 8 类服务

**位置**

- `database.py:40-42`
- `aemo_nem_scraper.py:201-211`
- `server.py:519-530`
- `web/src/components/FcasAnalysis.jsx`

**已验证事实**

- 代码已支持:
  - `raise1sec_rrp`
  - `lower1sec_rrp`
  - RFFR 别名映射
- 但现有 `trading_price_2025` / `trading_price_2026` 表结构仍只有 8 个 FCAS 列
- 后端 `FCAS_SERVICES` 仍只暴露 8 类
- 前端 `FcasAnalysis` 和 `RevenueStacking` 也只展示 8 类

**影响**

- 系统处于“采集口径想升级，展示口径未升级”的中间态。
- 新旧口径并存，后续容易造成历史回归和统计解释问题。

**建议**

- 统一完成一次 schema 升级与历史表迁移。
- 明确前端是否展示 8 类或 10 类，并提供口径说明。

### [P2] BESS 回测引擎比文档描述更先进，但仍存在乐观偏差

**位置**

- `bess_backtest.py:96-210`

**已验证事实**

- 当前引擎不是简单“两窗口均价差”，而是 MILP。
- 但仍存在以下简化:
  - 没有期末 SoC 约束
  - 每月独立优化，月末状态不传递
  - 每日循环上限约束被注释掉

**影响**

- 比静态窗口法真实，但仍可能高估收益。
- 作为投资基准时，仍应视为偏乐观版本。

**建议**

- 增加终端 SoC 约束
- 恢复或重构吞吐量/循环上限约束
- 在跨月边界上保持状态连续

| `dispatchCaps` | 否 | 否 | 否 | 否 | 出力上限 |
| `trapeziumAdjustments` | 否 | 否 | 否 | 否 | FCESS 梯形参数调整 |
| `definedContingency` | 否 | 否 | 否 | 否 | 定义的 contingency 集 |
| `definedLoadContingency` | 否 | 否 | 否 | 否 | 负荷 contingency 集 |
| `contingencySolution` | 否 | 否 | 否 | 否 | 惯量/应急求解结果 |
| `rocofControlRequirements` | 否 | 否 | 否 | 否 | RoCoF 控制需求 |
| `fcessMarketClearingPriceCeilings` | 否 | 否 | 否 | 否 | FCESS 价格上限 |
| `runIterationInformation` | 否 | 否 | 否 | 否 | 运行迭代元数据 |

### 7.4 结论

WEM ESS 原始数据中，当前真正被系统使用的只是一小部分价格字段。

而真正有助于把“价格”升级为“收益”的关键字段，当前全部未使用:

- `availableQuantities`
- `inServiceQuantities`
- `marketServiceRequirements`
- `marketShortfalls`
- `dispatchTotal`
- `priceSetting`
- `constraints`

## 9. 最终结论

当前系统最大的风险不是代码“跑不起来”，而是:

- 会跑
- 会出图
- 会给出结论
- 但这些结论里有一部分并不满足真实市场收益与调度约束

因此，现阶段最重要的不是继续堆更多图表，而是先把下面四件事做扎实:

1. 统一价格与收益口径
2. 接通 WEM ESS 全量数据链路
3. 把 WEM 与 NEM 的频率/时间轴彻底分开
4. 把投资分析从“演示模型”升级为“可解释、可审计、可做敏感性分析”的模型

做到这一步之后，平台才适合从“数据展示工具”真正升级成“储能市场分析工具”。
