# 商业化改造执行任务书

> 基于 `docs/global_grid_analysis_commercial_improvement_report.md` 与当前仓库现状整理。  
> 目标：把“商业化改进建议”压缩为当前项目可以执行的分阶段 backlog。  
> 文档日期：2026-04-27

---

## 1. 结论

`global_grid_analysis_commercial_improvement_report` 的总体方向是合理的，但它属于“目标蓝图”，不能直接当成单一开发计划执行。

原因不是建议有问题，而是当前项目现实边界很清楚：

- 当前后端仍是分析型 API 为主，尚未形成 SaaS 产品层。
- 当前数据层仍以 SQLite + 脚本同步为主，适合单机分析，不适合直接承载多市场商业平台。
- 当前前端仍是澳洲主工作台 + Fingrid 独立页的双入口结构，不是统一全球工作台。
- 当前 WEM、Fingrid、预测、投资分析已经具备雏形，但大量结果仍处于 preview / analytical-support 等级。

因此，正确执行方式不是“一次性做商业化平台”，而是拆成 3 个阶段、8 个工作流、若干可独立交付的任务包。

---

## 2. 执行原则

### 2.1 先做可信度，再做复杂度

优先顺序必须是：

1. 数据可信度和结果元数据
2. BESS 回测底座
3. 统一市场数据模型
4. 收益栈/筛选/报告等产品功能
5. 多租户、计费、企业安全

### 2.2 不把 preview 结果包装成 investment-grade

短期内必须在 UI、API、文档三处显式区分：

- `preview`
- `analytical`
- `analytical-preview`
- `investment-grade`（当前大部分模块还不应使用）

### 2.3 不做“大一统重构”

当前仓库不适合直接做全局推倒重来。应采用：

- 先补底层契约
- 再补核心引擎
- 再加产品化壳层
- 最后再升级部署架构

### 2.4 每个阶段都要有可验收产物

每个阶段至少要产出其中一种：

- 新 API
- 新表结构
- 新文档
- 新测试
- 新 UI 模块
- 新任务编排能力

---

## 3. 总体阶段划分

| 阶段 | 时间 | 目标 | 核心成果 |
|---|---:|---|---|
| P0 | 1-2 个月 | 把“分析结果”变成“可解释、可标级、可复用的结果” | 数据质量、统一元数据、BESS 回测 v1、WEM 标级 |
| P1 | 3-6 个月 | 把“工作台”变成“可交付分析产品” | canonical schema、WEM 补链路、Finland 扩展、筛选/告警/报告 |
| P2 | 6-12 个月 | 把“单机系统”变成“商业平台底座” | PostgreSQL/Parquet、queue/worker、多租户、可观测性 |

---

## 4. 工作流拆分

本项目建议拆成 8 个工作流并行治理：

1. `WF-A` 数据质量与结果可信度
2. `WF-B` BESS 回测与收益建模
3. `WF-C` FCAS / ESS 联合收益模型
4. `WF-D` 全球 canonical schema 与数据连接器
5. `WF-E` 产品层能力：筛选、告警、报告
6. `WF-F` API 契约与外部产品化
7. `WF-G` 架构升级：队列、存储、调度
8. `WF-H` 测试、治理与验收

## 11. 最终判断

这份商业化建议报告是合理的，尤其在以下判断上是正确的：

- 当前系统应从“图表展示”升级到“收益与风险解释”
- BESS Revenue Analytics 应成为产品主线
- WEM 当前只能视作 preview / analytical-preview
- Fingrid 不应长期孤立存在
- SQLite + 内置调度不应被当成最终商业架构

但真正可执行的工程路径，必须是：

> **先把结果做可信，再把模型做成体系，再把体系包装成产品，最后再把产品升级成平台。**

这就是本任务书的分解逻辑。
