# QA测试执行文档设计说明

> 日期：2026-04-22
> 状态：draft
> 目的：为当前仓库编写一份面向“开发 + 测试协作”的 QA 测试执行文档，覆盖真实存在的模块、真实可执行的测试入口、真实已知的测试缺口，不把理想化测试体系写进正文。

## 1. 这次要交付的是什么

这次要写的不是泛化的 QA 模板，也不是面向管理层的质量汇报材料。

这次真正要交付的是：

- 一份单文件的 Markdown QA 执行文档
- 主要服务对象是开发和测试协作执行
- 内容严格基于当前仓库现状
- 能直接指导环境准备、功能测试、接口测试、脚本验证、回归测试和缺陷记录

目标文档路径定为：

`docs/QA测试执行文档.md`

## 2. 文档的使用对象与使用场景

### 2.1 目标读者

- 开发人员
- 测试人员
- 参与联调、回归和发布验收的协作成员

### 2.2 使用场景

- 新人接手仓库后的测试准备
- 日常功能回归
- 版本发布前验证
- 问题复现与缺陷记录
- 自动化现状盘点和人工补测

### 2.3 不服务的场景

- 管理层质量周报
- 流程制度汇报
- 未来理想化测试体系蓝图
- 长期测试平台建设方案

这些不进入正文，只在必要处作为“当前缺口”简短说明。

## 3. 当前仓库现实基础

基于仓库现状，测试文档必须围绕以下现实来写：

- 项目是前后端分离结构：
  - `backend/` 提供 FastAPI API 与业务分析逻辑
  - `web/` 提供 React + Vite 单页分析工作台
  - `scrapers/` 提供 NEM/WEM/事件数据抓取脚本
  - `tests/` 与 `web/src/lib/*.test.js` 提供部分自动化测试资产
- 前端构建当前可通过，但 lint 不通过
- 前端 Node 原生测试可手动执行
- Python `unittest` 默认无法直接稳定跑通，存在导入路径和环境依赖问题
- 数据同步依赖真实外部源，部分测试天然属于“有限可信验证”

因此，正文必须同时满足两点：

1. 能指导人工测试执行
2. 能真实记录当前自动化测试现状，而不是假设自动化已经完整可用

## 4. 三种写法对比

### 方案 A：纯测试计划型文档

只写范围、策略、矩阵和回归清单，不写详细步骤。

优点：

- 文档短
- 适合快速汇总

缺点：

- 执行性不足
- 测试人员仍需要自己补步骤

### 方案 B：执行版文档，推荐

主体包含：

- 测试目标
- 环境要求
- 测试矩阵
- 模块级详细测试用例
- 自动化现状
- 回归清单
- 缺陷等级和验收标准

优点：

- 开发和测试可以直接照着执行
- 贴合当前仓库状态
- 最适合当前项目的成熟度

缺点：

- 文档会较长

### 方案 C：现状 + 理想未来双层文档

在执行文档中同时展开未来理想测试体系。

优点：

- 看起来更系统

缺点：

- 容易削弱执行文档的可操作性
- 会把“现在怎么测”写散

### 推荐结论

采用 `方案 B：执行版文档`。

正文只写当前仓库怎么测，不把未来体系做成正文主线。

## 5. 文档范围定义

### 5.1 测试覆盖范围

正文覆盖以下对象：

- 前端工作台
  - 首页加载与基础状态
  - 年份/区域/月份/季度/日类型筛选联动
  - 价格趋势
  - 事件叠加
  - 电网预测
  - 峰谷套利分析
  - FCAS 分析
  - BESS 模拟
  - 收入叠加
  - 充电窗口
  - 循环成本
  - 投资分析
  - 数据同步按钮与错误提示
- 后端 API
  - `/api/years`
  - `/api/summary`
  - `/api/price-trend`
  - `/api/event-overlays`
  - `/api/grid-forecast`
  - `/api/grid-forecast/coverage`
  - `/api/network-fees`
  - `/api/peak-analysis`
  - `/api/hourly-price-profile`
  - `/api/fcas-analysis`
  - `/api/investment-analysis`
  - `/api/sync_data`
- 数据脚本
  - NEM 抓取
  - WEM 抓取
  - WEM ESS slim 抓取
  - 事件同步脚本
  - 初始化/辅助脚本
- 自动化测试资产
  - Python `unittest`
  - 前端 `node:test`
  - 前端 `build/lint`

### 5.2 非覆盖范围

正文明确不纳入：

- 性能压测
- 安全渗透测试
- 完整浏览器兼容矩阵
- 移动端专项测试
- 生产环境灰度验证
- 面向未来的理想测试平台方案

## 6. 文档结构设计

目标文档 `docs/QA测试执行文档.md` 推荐使用以下结构：

1. 文档目标与适用范围
2. 项目测试对象说明
3. 测试环境与前置条件
4. 测试策略与执行方式
5. 测试范围与非测试范围
6. 模块测试矩阵
7. 详细测试用例
8. 自动化测试现状
9. 已知测试缺口与质量风险
10. 回归测试清单
11. 缺陷等级与提交流程
12. 发布前验收标准

该结构遵循“先总则，后矩阵，再详细执行，最后收口”的顺序。

## 7. 详细测试用例设计原则

### 7.1 用例粒度

详细测试用例要写到“可直接执行”，但不写成截图式培训手册。

每条用例统一采用以下字段：

- 用例编号
- 所属模块
- 优先级
- 测试类型
- 前置条件
- 测试步骤
- 输入/操作
- 预期结果
- 结果记录
- 备注/风险

### 7.2 优先级

- `P0`
  - 主链路不可用
  - 阻断系统基本使用
- `P1`
  - 核心分析模块结果异常
  - 主要功能不可用或明显错误
- `P2`
  - 边界问题、体验问题、次要模块异常

### 7.3 正文详细展开的模块

正文完整详细用例重点放在：

- 前端总入口与筛选联动
- 价格趋势
- 事件叠加
- 电网预测
- 峰谷套利
- FCAS 分析
- 投资分析
- 数据同步与抓取脚本
- 核心回归主路径

支撑型接口如 `years`、`summary`、`network-fees` 可放入矩阵和精简用例，不单独膨胀为长章节。

## 8. 自动化测试现状章节设计

这一章必须写“当前事实”，不能写成理想状态。

### 8.1 后端自动化

需要明确：

- 当前存在 `tests/` 目录下的 `unittest`
- 覆盖事件叠加、电网预测、WEM ESS、若干修复项
- 默认执行存在导入路径依赖
- 实际运行还依赖 `fastapi`、`pulp` 等环境

### 8.2 前端自动化

需要明确：

- 当前存在 `web/src/lib/*.test.js`
- 使用 Node 原生 `node:test`
- 当前可手动执行通过
- 但未接入 `package.json` 标准测试脚本

### 8.3 工程校验现状

需要明确：

- `vite build` 当前可通过
- `eslint` 当前不通过
- 可构建不等于质量门禁通过

### 8.4 文档写法要求

每项自动化资产必须写清楚：

- 存在位置
- 执行命令
- 当前可执行性
- 当前结果
- 当前阻塞点

## 9. 风险与已知缺口写法

文档中需要单列“当前已知质量风险”，但不能写成泛泛风险库。

必须围绕当前仓库真实问题展开，例如：

- Python 测试环境依赖不稳定
- 前端 lint 未收敛
- 数据同步依赖外部源
- 部分分析模块更适合做“有限可信验证”
- 自动化未形成完整门禁

这一章的目标不是解决所有问题，而是防止测试执行者误判当前质量能力。

## 10. 发布前回归与验收标准

文档末尾需要提供一套最小可执行发布检查项：

- 服务可启动
- 前端可打开
- 关键筛选联动正常
- 核心 API 返回正常
- 价格趋势可展示
- 电网预测可返回结果
- FCAS 与套利模块可加载
- 投资分析可提交并返回结果
- 数据同步入口行为可验证

并定义：

- 哪些问题阻断发布
- 哪些问题可以带缺陷放行

## 11. 文档风格要求

最终 QA 执行文档必须满足以下风格约束：

- 中文为主，术语保留必要英文
- 面向执行，不写空泛理论
- 用表格呈现矩阵、环境要求、自动化现状
- 用列表呈现详细测试用例、回归清单、缺陷等级
- 所有结论都应能在当前仓库找到对应依据

## 12. 实施边界

本设计说明只定义 QA 文档的结构、范围和写法。

后续实际编写 `docs/QA测试执行文档.md` 时，必须继续坚持：

- 基于当前仓库现状
- 不把未来理想测试体系写入正文
- 不假设不存在的 CI、测试平台或质量门禁已经存在
- 对“能跑”“能构建”“能回归”“能验证”的表述保持严格区分
