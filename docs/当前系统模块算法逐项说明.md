## 1. 系统总览

当前系统本质上是一个“澳洲电力市场分析与解释系统”，不是自动交易系统，也不是投资级收益预测器。它由 4 层组成：

1. 数据采集层  
   从 AEMO NEM、AEMO WEM、WEM dispatch solution、AEMO/BOM 官方事件源抓数。

2. 数据存储层  
   将数据写入 SQLite；部分重型结果再写入 SQLite 分析缓存与 Redis 响应缓存。

3. 后端分析层  
   用 `server.py` 提供 API，对价格、套利、FCAS、投资测算、事件解释、电网预测做计算。

4. 前端展示层  
   `web/src/App.jsx` 负责全局筛选和模块装配，各图表模块负责调用各自接口并做二次整理展示。

## 2. 全局共用规则

### 2.1 时间筛选规则

后端多个接口共用 `_build_temporal_filters()`，规则如下：

- `month` 优先于 `quarter`
- `day_type` 会在 `month`/`quarter` 基础上继续叠加
- `month` 用 `YYYY-MM-%` 过滤
- `quarter` 用月份集合过滤：
  - `Q1 = 01,02,03`
  - `Q2 = 04,05,06`
  - `Q3 = 07,08,09`
  - `Q4 = 10,11,12`
- `day_type`
  - `WEEKDAY = 周一到周五`
  - `WEEKEND = 周六周日`

注意：

- `price-trend`、`peak-analysis`、`fcas-analysis`、`event-overlays` 都支持完整的 `month / quarter / day_type`
- `hourly-price-profile` 目前只支持 `month`
- `InvestmentAnalysis` 和 `BessSimulator` 前端故意不跟随这些子时段筛选，按全年模型运行

### 2.2 市场分支规则

- `NEM`：主数据来自 `trading_price_YYYY`
- `WEM`：
  - 能量价格来自 WEM 交易价格数据
  - FCAS/约束相关来自 WEM ESS slim 表

### 2.3 缓存规则

当前系统有两层缓存：

1. Redis 响应缓存  
   用于高频读接口，例如：
   - `/api/price-trend`
   - `/api/peak-analysis`
   - `/api/fcas-analysis`
   - `/api/hourly-price-profile`
   - `/api/event-overlays`
   - `/api/grid-forecast`
   - `/api/investment-analysis`

2. SQLite 分析缓存  
   主要用于投资分析内部的重计算结果：
   - 单年套利回测
   - FCAS 基线估算
   - 投资分析最终响应

另外，投资分析还做了“同 key 并发去重”：

- 若多个请求同时计算同一组参数，只让一个线程计算
- 其余线程等待 leader 线程结果

## 3. 数据采集模块

## 3.1 NEM 历史价格采集：`aemo_nem_scraper.py`

### 作用

抓取 NEM 历史 `TRADINGPRICE`，可选抓取 `DISPATCHPRICE` 里的 FCAS 字段，并写入 `trading_price_YYYY`。

### 核心算法

1. 先按年月枚举归档目录
2. 对每个月尝试多个 ZIP 命名模板
3. 若猜测 URL 失败，再扫描目录页寻找真实文件名
4. 下载 ZIP 后解析 CSV
5. MMSDM CSV 解析规则：
   - `C` 行忽略
   - `I` 行作为列头
   - `D` 行作为数据
6. 对每条记录提取：
   - `settlement_date`
   - `region_id`
   - `rrp_aud_mwh`
7. 若启用 `--fcas`，则额外抓 `DISPATCHPRICE`，并映射 FCAS 字段：
   - `raise1sec_rrp`
   - `raise6sec_rrp`
   - `raise60sec_rrp`
   - `raise5min_rrp`
   - `raisereg_rrp`
   - `lower1sec_rrp`
   - `lower6sec_rrp`
   - `lower60sec_rrp`
   - `lower5min_rrp`
   - `lowerreg_rrp`

### 特点

- 这是“原始历史行情层”
- 后续绝大多数 NEM 分析模块都依赖这个表

## 3.2 WEM 交易价格采集：`aemo_wem_scraper.py`

### 作用

抓取 WEM `ReferenceTradingPrice`，写入同一套价格表，区域固定为 `WEM`。

### 核心算法

1. 优先尝试 `previous/ReferenceTradingPrice_YYYYMMDD.zip`
2. 若没有历史 ZIP，再退回 `current/ReferenceTradingPrice_YYYY-MM-DD.json`
3. 解析 `tradingInterval` 和 `referenceTradingPrice`
4. 将时间统一标准化为 `AWST`
5. 写入：
   - `settlement_date`
   - `region_id = WEM`
   - `rrp_aud_mwh`

### 特点

- 保存的是 WEM 原始交易价格
- 时间与 NEM 不同，WEM 是 `AWST`
- 不做 30 分钟到 5 分钟的上采样；分析层自己处理

## 3.3 WEM ESS slim 采集：`aemo_wem_ess_scraper.py`

### 作用

抓取 WEM dispatch solution，但不保留全量大 JSON，只提取 ESS 分析需要的“瘦字段”。

### 核心算法

对每个 dispatch solution，只提取以下几类字段：

1. 价格
   - `energy_price`
   - `regulation_raise_price`
   - `regulation_lower_price`
   - `contingency_raise_price`
   - `contingency_lower_price`
   - `rocof_price`

2. 可用量 / 在服量 / 需求量 / 短缺量 / 出清量
   - `availableQuantities`
   - `inServiceQuantities`
   - `marketServiceRequirements`
   - `marketShortfalls`
   - `dispatchTotal`

3. 限价信息
   - `priceSetting[].isMarketServiceCapped`

4. 约束摘要
   - `binding_count`
   - `near_binding_count`
   - `binding_max_shadow_price`
   - 各类约束最大 shadow price

### 约束摘要算法

遍历 `constraints`：

- 若 `bindingConstraintFlag = true`
  - `binding_count += 1`
  - 更新 `binding_max_shadow_price = max(abs(shadowPrice))`
  - 按 `constraintType` 更新各类型最大 shadow price
- 若 `nearBindingConstraintFlag = true`
  - `near_binding_count += 1`
  - 更新 `near_binding_max_shadow_price`

### 特点

- 这是一个“轻量预览层”，不是 WEM 全量市场结果库
- 后续 WEM FCAS 分析和 WEM Grid Forecast 都依赖这个 slim 表

## 3.4 电网事件采集：`aemo_grid_event_scraper.py`

### 作用

调用 `grid_events.sync_event_sources()`，把官方事件源同步到 SQLite。

### 采集源

- NEM Market Notices
- NEM High Impact Outages
- WEM Dispatch Advisories
- WEM Realtime Outages
- BOM Warnings

### 输出对象

- `grid_event_raw`
- `grid_event_state`
- `grid_event_sync_state`

## 4. 核心底层引擎

## 4.1 事件归一化引擎：`grid_events.py`

### 作用

把不同官方源的原始事件统一归一成系统可消费的“事件状态层”。

### 4.1.1 原始事件到标准状态的映射

系统固定 6 类标准状态：

- `reserve_tightness`
- `security_intervention`
- `network_stress`
- `supply_shock`
- `demand_weather_shock`
- `post_event_structural`

### 4.1.2 分类算法

函数 `_classify_state_types(text)` 通过关键词规则映射：

- 包含 `LOR1/LOR2/LOR3/RESERVE NOTICE` → `reserve_tightness`
- 包含 `DIRECTION/INTERVENTION/SYSTEM SECURITY` → `security_intervention`
- 包含 `OUTAGE/INTERCONNECTOR/NETWORK/CONSTRAINT` → `network_stress`
- 包含 `GENERATOR/UNIT/TRIP/FACILITY` → `supply_shock`
- 包含 `HEATWAVE/FIRE WEATHER/BUSHFIRE/STORM/DEMAND` → `demand_weather_shock`
- 包含 `REPORT/REVIEW/POST EVENT` → `post_event_structural`

一个原始事件可以映射到多个状态。

### 4.1.3 严重度算法

函数 `_normalize_severity(text)` 通过关键词给严重度：

- `high`
  - `LOR3`
  - `LOAD SHEDDING`
  - `BLACK SYSTEM`
  - `EMERGENCY`
  - `LOR2`
  - `INTERVENTION`
  - `SYSTEM SECURITY`
- `medium`
  - `LOR1`
  - `OUTAGE`
  - `CONSTRAINT`
  - `SEVERE WEATHER`
  - `HEATWAVE`
- 其他 → `low`

### 4.1.4 置信度算法

按 source 固定打分：

- `nem_high_impact_outage` / `wem_realtime_outage` / `bom_warnings` → `0.9`
- `wem_dispatch_advisory` → `0.85`
- 其他 → `0.75`

### 4.1.5 区间合并算法

`merge_explanation_states()` 会把相邻同类状态合并，条件是：

- 同一 `market`
- 同一 `region`
- 同一 `state_type`
- 同一 `severity`
- 两段状态之间时间间隔 `<= 360 分钟`

合并时：

- `end_time` 取最大
- `confidence` 取更高值
- `impact_domains` 取并集
- `evidence_event_ids` 取并集

### 4.1.6 日汇总算法

`_build_daily_rollup()` 会把状态扩展到其覆盖的每一天，然后：

- 按 `state_type` 计数
- 同类取最高 severity
- 再按“severity 高低 + 次数多少”排序，得到 `top_states`

### 4.1.7 覆盖度算法

`_coverage_quality()`：

- NEM：
  - 3 个关键源都 `ok` → `full`
  - 部分 `ok` → `partial`
  - 都没有 → `none`
- WEM：
  - 只要核心源可用 → `core_only`
  - 否则 `none`

## 4.2 电网预测引擎：`grid_forecast.py`

### 作用

给 NEM/WEM 生成未来 `24h / 7d / 30d` 的“风险与机会信号”，不是 ML 预测，也不是收益预测。

### 4.2.1 最近市场特征构造

`build_recent_market_features()`

#### NEM

从最近 `288` 个 interval 读取数据，约等于最近 24 小时，计算：

- `recent_avg_price`
- `recent_max_price`
- `recent_min_price`
- `negative_ratio`
- `recent_fcas_avg`

其中：

- `negative_ratio = 负价 interval 数 / 总 interval 数`
- `recent_fcas_avg = 所有可用 FCAS 列加总后的平均值`

#### WEM

从 WEM ESS slim 最近 `96` 个 interval 读取：

- `recent_avg_price`
- `recent_max_price`
- `binding_count_avg`
- `binding_shadow_max`
- `network_shadow_max`
- `shortfall_total`
- `recent_fcas_avg`

### 4.2.2 事件特征构造

`build_event_features()`

读取未来预测窗口内与该区域重叠的事件状态，计算：

- `event_count`
- `state_types`
- `severity_score`

严重度积分固定为：

- `low = 15`
- `medium = 30`
- `high = 45`

### 4.2.3 NEM 24h 预测算法

`build_nem_24h_forecast()`

数据来源：

- 最近市场历史
- NEM predispatch
- 事件状态层

先从 predispatch 取未来窗口的：

- `max_future_price`
- `min_future_price`
- `max_future_demand`

然后计算分数：

- `spike_score = clamp(max_future_price / 500 * 100 + min(severity_score, 20))`
- `negative_score = clamp(max(abs(min(min_future_price, 0)) * 1.5, recent_negative_ratio * 100))`
- `reserve_score = clamp(25 + recent_fcas_avg * 0.4 + severity_score)`
- `fcas_score = clamp(reserve_score * 0.75 + recent_fcas_avg * 0.6)`
- `charge_score = clamp(negative_score + (10 if min_future_price < 0 else 0))`
- `discharge_score = clamp(spike_score + (10 if max_future_price > 300 else 0))`
- `grid_stress = clamp(max(spike_score, reserve_score, recent_avg_price * 0.25, max_future_demand / 180))`

其中 `clamp()` 把值限制在 `0~100`。

#### 窗口生成规则

遍历未来 predispatch 点：

- 若 `price < 0`，生成 `charge` window
- 若 `price > 150`，生成 `discharge` window

因此 24h 模式是“阈值触发式窗口标记”，不是连续最优化排程。

### 4.2.4 NEM 7d / 30d 预测算法

`build_nem_long_horizon_forecast()`

这个模块不做逐点价格预测，而是做 regime outlook。

关键估算：

- `projected_min`
  - 由最近低价、负价比例、horizon 放大系数共同决定
- `projected_max`
  - 由最近高价、均价、价带宽度、事件严重度共同放大得到

分数：

- `spike_score`
- `negative_score`
- `reserve_score`
- `fcas_score`
- `charge_score = negative_score + 6`
- `discharge_score = spike_score + 8`
- `grid_stress = max(spike_score, reserve_score, recent_avg_price * 0.4 + severity_score * 0.35)`

输出只有一个 `core_risk_window`，表示整个未来区间的结构性判断。

### 4.2.5 WEM 核心预测算法

`build_wem_core_forecast()`

基于 WEM ESS slim + 事件层，算的是核心约束/短缺压力，不是全市场预测。

分数公式：

- `grid_stress = clamp(25 + binding_count_avg * 10 + binding_shadow_max / 10 + shortfall_total * 4 + severity_score * 0.6)`
- `spike_score = clamp(recent_max_price / 4 + binding_shadow_max / 8)`
- `negative_score = 0`
- `reserve_score = clamp(20 + shortfall_total * 8 + recent_fcas_avg * 0.05)`
- `fcas_score = clamp(30 + recent_fcas_avg * 0.1 + binding_count_avg * 8)`
- `charge_score = clamp(15 + network_shadow_max * 0.08)`
- `discharge_score = clamp(spike_score + 10)`

输出一个 `core_risk_window`，并明确：

- `coverage_quality = core_only`
- `investment_grade = false`

### 4.2.6 快照缓存算法

`get_grid_forecast_response()`

会先按：

- `market`
- `region`
- `horizon`
- `as_of_bucket`

查 SQLite snapshot cache。

bucket 规则：

- `24h`：按整点 bucket
- `7d/30d`：按整日 bucket

过期时间：

- `24h`：1 小时
- `7d`：6 小时
- `30d`：12 小时

## 4.3 套利回测引擎：`bess_backtest.py`

### 作用

对某年某区域的现货价格做电池套利上限回测。

### 核心口径

这是一个：

- `optimized_hindsight`
- `physical_upper_bound`

模型，不代表真实可实现收益。

### 4.3.1 月度 MILP 建模

按月份分开求解，每个月建立一个线性规划：

变量：

- `charge[t]`
- `discharge[t]`
- `soc[t]`
- `charge_on[t]` 二元
- `discharge_on[t]` 二元

约束：

1. SoC 递推

`soc[t] = soc[t-1] + charge[t] * dt * eta - discharge[t] * dt / eta`

其中：

- `eta = sqrt(round_trip_efficiency)`

2. 不允许同一时段同时充放

- `charge_on[t] + discharge_on[t] <= 1`
- `charge[t] <= power_mw * charge_on[t]`
- `discharge[t] <= power_mw * discharge_on[t]`

3. 月初月末 SoC 固定

- 每个月 `initial_soc = 50% capacity`
- 强制 `terminal_soc = 50% capacity`

4. 月度吞吐约束

`sum(charge[t] * dt) <= cycles_per_day * month_days * capacity_mwh`

默认：

- `ROUND_TRIP_EFFICIENCY = 0.87`
- `CYCLES_PER_DAY = 1`

### 4.3.2 目标函数

最大化：

`sum((discharge[t] - charge[t]) * price[t] * dt)`

也就是理想情况下只看价差的套利毛收益。

### 4.3.3 输出

- `total_revenue_aud`
- `revenue_per_mw_year`
- `monthly`
- `trading_days`
- `annual_charge_mwh`
- `annual_discharge_mwh`
- `throughput_limit_mwh`
- `initial_soc_mwh`
- `terminal_soc_mwh`

### 默认值

- `NSW1 = 45`
- `QLD1 = 42`

- `SA1 = 55`
- `TAS1 = 48`
- `WEM = 40`

### 作用方式

## 5.4 `/api/price-trend`

### 作用

输出价格时序、摘要统计、负价小时分布。

### 输入

- `year`
- `region`
- `month?`
- `quarter?`
- `day_type?`
- `limit`

### 计算流程

1. 根据 `_build_temporal_filters()` 过滤目标样本
2. 统计总行数 `total_rows`
3. 读取全部时序数据并按时间排序
4. 若点数 `> limit`，则用 `lttbc.downsample()` 做 LTTB 降采样
5. 再做一遍聚合 SQL 生成统计指标

### 核心统计指标

- `min`
- `max`
- `avg`
- `neg_ratio = 负价点数 / 总点数 * 100`
- `neg_avg`
- `neg_min`
- `pos_avg`
- `pos_max`
- `days_below_100`
  - `rrp < -100` 的自然日数
- `days_above_300`
  - `rrp > 300` 的自然日数

### 负价小时分布算法

对 `rrp_aud_mwh < 0` 的样本按小时聚合：

- `hour_bucket = substr(datetime(settlement_date, '-1 second'), 12, 2)`

然后补足 `00~23` 的所有小时。

### 输出

- `stats`
- `advanced_stats`
- `hourly_distribution`
- `data`

## 5.5 `/api/peak-analysis`

### 作用

做滑窗峰谷价差分析，是系统里套利分析的核心底座之一。

### 输入

- `year`
- `region`
- `aggregation = daily / weekly / monthly / yearly`
- `network_fee?`
- `month?`
- `quarter?`
- `day_type?`

### 5.5.1 窗口定义

窗口大小来自 `network_fees.get_window_sizes(region)`：

- NEM 5 分钟市场：
  - `1h = 12`
  - `2h = 24`
  - `4h = 48`
  - `6h = 72`
- WEM 30 分钟市场：
  - `1h = 2`
  - `2h = 4`
  - `4h = 8`
  - `6h = 12`

### 5.5.2 每日峰谷计算算法

先按日分组，每天得到一个价格数组。

对每个窗口长度 `w_size`：

1. 先算首个窗口和
2. 再用 running sum 滑动更新
3. 维护：
   - `best_max`
   - `best_min`
4. 转成窗口均价：
   - `peak = best_max / w_size`
   - `trough = best_min / w_size`

### 5.5.3 价差与净价差

仅对 `2h / 4h / 6h` 输出 spread：

- `spread = peak - trough`
- `net_spread = spread - 2 * network_fee`

这里默认把网络费视为“充一次 + 放一次”的双边费用。

### 5.5.4 聚合算法

如果不是 `daily`，则把每日结果继续按：

## 5.7 `/api/fcas-analysis`

这个接口对 NEM 和 WEM 走完全不同的算法。

### 5.7.1 NEM 分支

#### 输入

- `year`
- `region`
- `aggregation = daily / weekly / monthly`
- `capacity_mw`
- `month?`
- `quarter?`
- `day_type?`

#### 步骤 1：识别可用 FCAS 列

先用 `PRAGMA table_info` 看表里有哪些 FCAS 列，只分析存在的列。

如果：

- 没有 FCAS 列 → 返回 `has_fcas_data = false`
- 列存在但全是 null → 返回 `has_fcas_data = false`

#### 步骤 2：服务分解

对每个 FCAS 服务计算：

- `avg_price = AVG(col)`
- `max_price = MAX(col)`
- `est_revenue_k = avg_price * capacity_mw * total_intervals * (5/60) / 1000`

这里是一个非常直接的 price-taker 粗估：

- 假设每个 interval 都以 `capacity_mw` 完整参与
- 每个 interval 时长固定为 `5/60` 小时

#### 步骤 3：小时分布

定义：

`total_fcas_expr = 所有可用 FCAS 列 COALESCE 后求和`

然后按小时聚合：

- `avg_total_fcas = AVG(total_fcas_expr)`

#### 步骤 4：时间序列

按 `daily / weekly / monthly` 聚合，输出：

- 每个 FCAS 服务的平均价格
- `total_fcas_avg`
- `intervals`

#### 步骤 5：summary

- `total_avg_fcas_price = 各服务 avg_price 之和`
- `total_est_revenue_k = 各服务 est_revenue_k 之和`
- `data_points_with_fcas`
- `capacity_mw`

### 5.7.2 WEM 分支

WEM 分支实际调用 `_get_wem_ess_analysis()`。

#### 数据来源

- `WEM_ESS_MARKET_TABLE`
- `WEM_ESS_CONSTRAINT_TABLE`

#### 服务集合

- `regulation_raise`
- `regulation_lower`
- `contingency_raise`
- `contingency_lower`
- `rocof`

#### capture 估算核心：`_estimate_wem_capture()`

对每个服务、每个 interval：

1. `candidate_mw = min(capacity_mw, max(dispatch_total + shortfall, dispatch_total, 0))`
2. `capture_rate`
   - 若 `in_service > 0`
     - `capture_rate = min(dispatch_total / in_service, 1)`
   - 否则若 `dispatch_total > 0`
     - `capture_rate = 1`
   - 否则 `0`
3. `enabled_mw = min(candidate_mw * capture_rate, max(dispatch_total + shortfall, 0))`
4. `tightness`
   - 若 `requirement > 0`
     - `tightness = max(0, 1 - min(available / requirement, 1))`

#### 收入估算

对每个 interval：

`revenue = enabled_mw * price * interval_hours / 1000`

当前代码用：

- `interval_hours = 5/60`

#### 统计输出

每个服务输出：

- `avg_price`
- `max_price`
- `est_revenue_k`
- `avg_requirement_mw`
- `avg_in_service_mw`
- `avg_available_mw`
- `avg_dispatch_total_mw`
- `avg_capture_rate`
- `avg_tightness`
- `shortfall_intervals`
- `capped_intervals`

整体 summary 还输出：

- `coverage_days`
- `preview_mode`
  - `1 天 = single_day_preview`
  - `>1 天 = multi_day_preview`
- `investment_grade = false`

### 结论

NEM FCAS 是“历史价格 x 理想容量”的粗估。  
WEM FCAS 是“slim 表字段驱动的 preview 估算”，更贴近 dispatch 字段，但仍不是投资级结果。

## 5.11 `/api/investment-analysis`

### 作用

做项目级投资测算，是系统里最重的分析接口。

### 输入参数

来自 `InvestmentParams`，主要包括：

- 储能参数
  - `power_mw`
  - `duration_hours`
  - `round_trip_efficiency`
  - `degradation_rate`
- 成本参数
  - `capex_per_kwh`
  - `fixed_om_per_mw_year`
  - `variable_om_per_mwh`
  - `grid_connection_cost`
  - `land_lease_per_year`
- 财务参数
  - `discount_rate`
  - `project_life_years`
- 收入参数
  - `revenue_capture_rate`
  - `fcas_revenue_per_mw_year`
  - `fcas_revenue_mode`
  - `capacity_payment_per_mw_year`
  - `backtest_years`

### 5.11.1 总体流程

1. `capacity_mwh = power_mw * duration_hours`
2. 查 Redis 响应缓存
3. 查 SQLite 分析缓存
4. 若未命中，执行 `_compute_investment_analysis_response()`

### 5.11.2 套利基线

对每个 `backtest_year`：

1. 调 `bess_backtest.backtest_arbitrage()`
2. 得到该年：
   - `gross_arbitrage`
   - `per_mw`
   - `trading_days`
3. 多年平均得到 `avg_gross`
4. 套利可实现基线：

`baseline_arbitrage = avg_gross * revenue_capture_rate`

也就是：

- 回测先给“优化上限”
- 再乘 capture rate 折减成可实现基线

### 5.11.3 FCAS 基线

#### NEM auto

调用 `_estimate_nem_fcas_baseline()`：

对每个回测年：

- `annual_estimate = SUM(total_fcas_expr * power_mw * 5/60)`

有值则多年平均。

如果没拿到有效历史 FCAS：

- 自动回退到 manual

#### WEM auto

当前直接回退到 manual。

#### manual

`baseline_fcas = fcas_revenue_per_mw_year * power_mw`

### 5.11.4 容量收入

`baseline_capacity = capacity_payment_per_mw_year * power_mw`

当前完全手动输入，不从市场数据反推。

### 5.11.5 CAPEX / OPEX

#### CAPEX

`total_capex = capex_per_kwh * capacity_mwh * 1000 + grid_connection_cost`

#### 固定 OPEX

`annual_fixed_om = fixed_om_per_mw_year * power_mw + land_lease_per_year`

#### 可变 OPEX

`annual_var_om = variable_om_per_mwh * actual_est_annual_discharge`

其中：

`actual_est_annual_discharge = avg_annual_discharge_mwh * revenue_capture_rate`

### 5.11.6 现金流投影

每年退化因子：

`deg_factor = (1 - degradation_rate) ^ (year - 1)`

收入：

- `rev_arbitrage = baseline_arbitrage * deg_factor`
- `rev_fcas = baseline_fcas * deg_factor`
- `rev_capacity = baseline_capacity`

说明：

- 套利和 FCAS 随电池衰减
- 容量收入不衰减

OPEX：

`total_opex = annual_fixed_om + annual_var_om * deg_factor`

净现金流：

`net = total_rev - total_opex`

累计现金流：

`cumulative += net`

### 5.11.7 NPV / IRR / Payback / ROI

#### NPV

`npv = sum(cash_flow[i] / (1 + discount_rate)^i)`

#### IRR

`_compute_irr()` 使用二分法：

- 下界 `-0.5`
- 上界 `5.0`
- 迭代直到 `abs(npv) < tol`

#### Payback

累计现金流第一次大于等于 0 的年份。

#### ROI

`roi = 项目期内总净现金流 / total_capex`

### 5.11.8 当前口径结论

这个模块是“保守现金流模型”，但它的套利底座仍然来自 hindsight 优化上限，因此：

- 不能理解为真实历史实得收入
- 只能理解为“上限折减后的基线模型”

## 6.8 `BessSimulator.jsx`

### 作用

做一个“单循环利润瀑布图”，不是完整回测，也不是投资模型。

### 输入来源

它先请求：

`/api/peak-analysis?aggregation=yearly`

然后取：

- `summary.avg_spread_4h`

作为代表性毛价差。

### 瀑布图公式

设：

- `grossSpread = avg_spread_4h`

则依次扣减：

1. `rteLoss = grossSpread * (1 - rte)`
2. `auxLoss = grossSpread * auxPct`
3. `networkCost = networkFee * 2`
4. `mlfLoss = grossSpread * (1 - mlf)`
5. `aemoFeeCost = aemoFee * 2`
6. `degradation = degradationCost`

最后：

`netProfit = grossSpread - rteLoss - auxLoss - networkCost - mlfLoss - aemoFeeCost - degradation`

### 年化估算

`dailyRevenue = netProfit * capacityMw * durationH * cyclesPerDay`

`annualRevenue = dailyRevenue * 365`

### 结论

这是“代表性单循环 waterfall”，不是 backtest，也不是 cash-flow model。

## 6.11 `CycleCost.jsx`

`/api/peak-analysis?aggregation=daily`

### 前端算法

1. 优先取 `net_spread_4h`
2. 若不存在，再退回 `spread_4h`
3. 取所有日值做直方图，bucket size 固定为 `25`

4. 用户输入 `degradationCost`

   - `profitableDays = spread > degradationCost`
   - `unprofitableDays = totalDays - profitableDays`
   - `profitableRatio`
   - `avgSpread`
   - `maxSpread`

### 颜色阈值

- `rangeNum >= degradationCost` → 绿色
- `rangeNum >= degradationCost * 0.7` → 黄色
- 否则红色

### 结论

这是“日级净价差 vs 用户定义循环成本阈值”的经验判断器。

## 8. 当前实现的关键口径提醒

### 8.1 套利收益不是“真实历史实得”

投资模型与部分收益推导依赖 `bess_backtest.py`，而该模块是：

- 完美预知
- 月度 MILP
- 物理上限

所以只能视为：

- 理论调度上限
- 再经 `revenue_capture_rate` 折减后的可实现基线

### 8.2 WEM FCAS 不是全量投资级口径

WEM FCAS 当前基于 slim ESS 表和 dispatch 字段做 preview 估算：

- 可以解释市场状态
- 可以给出轻量收益预览
- 不适合直接用于项目融资或正式投资测算

### 8.3 Grid Forecast 不是机器学习模型

当前 forecast 完全是规则型信号引擎：

- 历史统计
- 官方 predispatch
- 事件状态
- 约束与 shadow price

没有训练模型，也没有做概率校准。

### 8.4 部分模块使用“代表性指标”而不是完整回测

例如：

- `BessSimulator` 用的是 `avg_spread_4h`
- `ChargingWindow` 用的是“平均小时价格”
- `CycleCost` 用的是“日级 net_spread_4h”

这些更适合解释和快速判断，不等于真实调度收益。

## 9. 一句话总结每个主模块

- `SummaryStats`：价格统计摘要面板
- `PriceChart`：价格曲线 + 事件浅色区间
- `GridForecast`：规则型电网前瞻信号面板
- `HourlyDistributionChart`：负价发生时段分布
- `PeakAnalysis`：滑窗峰谷价差分析
- `FcasAnalysis`：FCAS 服务拆分与粗收益估算
- `BessSimulator`：代表性单循环利润 waterfall
- `RevenueStacking`：套利与 FCAS 叠加展示
- `ChargingWindow`：按平均小时价格筛选充放电窗口
- `CycleCost`：用净价差对比循环损耗阈值
- `InvestmentAnalysis`：基于套利上限折减基线的项目现金流模型
